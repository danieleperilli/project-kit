## Generic Rules

### Formatting

- Follow the dominant formatting style already used by the repository.
- Keep indentation, quoting, and file organization consistent inside each language or subsystem.
- Prefer small, reviewable diffs over large style-only rewrites.

### Naming

- Reuse existing naming conventions before introducing a new one.
- Prefer descriptive names over abbreviations unless the codebase already standardizes them.

### Documentation

- Document non-obvious behavior, constraints, and edge cases.
- Keep code, comments, identifiers, and structured documentation in English unless the repository explicitly requires otherwise.

### Testing & Validation

- Run or describe the most relevant validation for the changed area.
- Call out residual risks or assumptions when validation is incomplete.

### Refinement

- Replace or refine this generic guide once the project adopts language-specific or team-specific rules.
